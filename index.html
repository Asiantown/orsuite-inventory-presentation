1 · Environment

The inventory environment

At each period, a policy observes net inventory and outstanding orders, then orders from a fast expensive supplier and a slower regular supplier before random demand is realized.

ObserveInventory and pipeline

Net stock or backlog plus orders already in transit.

ActChoose two quantities

[fast, regular], each from 0 through 4.

ArriveReceive due orders

Previously placed shipments advance through the pipeline.

DemandSample 0 through 4

Each integer demand value is equally likely.

ScoreCalculate total cost

Sourcing, holding, and backlog penalties.

Observation

Five conceptual state values

One fast-supplier pipeline slot, three regular-supplier pipeline slots, and encoded net inventory. The pipeline prevents the policy from forgetting orders already on the way.

Action

MultiDiscrete([5, 5])

The first integer is the fast order and the second is the regular order. Five choices for each supplier produce 25 feasible order pairs.

Episode

100 inventory periods

The task has no natural terminal state. Gymnasium reports terminated=False and truncates the episode after the 100-period horizon.

Uncertainty

Stochastic demand, fixed lead times

Demand is independent discrete-uniform from 0 through 4, with mean 2. Lead times are fixed at one and three periods.

Per-period cost Fast sourcing cost + holding cost on positive inventory + backlog cost on negative inventory. The environment returns the negative of this value as reward because reinforcement-learning algorithms maximize reward.

3 · Stable-Baselines3 algorithms

Two learning approaches, the same decisions

Stable-Baselines3 supplies both implementations. Each model trains against the same Gymnasium environment and can choose every feasible fast/regular order pair.

Policy-based

PPO

Learns probabilities for the fast and regular order quantities directly from MultiDiscrete([5, 5]). Clipped updates limit how far the policy changes after each rollout.

  • Native two-component action
  • Recent on-policy rollouts
  • Complete 512-step batches
Value-based

DQN

Predicts the long-run value of each complete order pair, reuses past experience from a replay buffer, and stabilizes targets with a separate target network.

  • One joint Discrete(25) action
  • Replay memory
  • Exploration over the first 30% of training

DQN action-wrapper equivalence

Stable-Baselines3 DQN accepts one discrete action, so a reversible wrapper enumerates the same 25 pairs available to PPO.

Index 0[0 fast, 0 regular]
Index 1[0 fast, 1 regular]
Index 24[4 fast, 4 regular]

Each model uses one uninterrupted learn() call. A callback evaluates checkpoints without restarting training, and every model sees the same held-out demand stream. PPO can slightly exceed a requested checkpoint because it finishes complete rollout batches, so the plot uses actual SB3 timesteps.

Exact algorithm and calibration settings

PPO

  • MlpPolicy, gamma 0.99
  • 512 rollout steps, batch size 64
  • 10 epochs, learning rate 3e-4

DQN

  • MlpPolicy, gamma 0.99
  • Learning rate 1e-4; starts after 1,000 steps
  • 50,000 replay buffer, batch size 64
  • Train every 4 steps; target update every 10,000

TBS calibration

  • Separate demand seed 1,100,000
  • Regular base search: 0 through 4
  • Fast target search: 0 through 20

Reference policy

The adapted tailored base-surge (TBS) reference orders 1 unit from the regular supplier each period and uses the fast supplier to move the near-term inventory position toward 7. Those two values are selected on a separate calibration stream before held-out evaluation. TBS is an inventory-specific heuristic, not a third machine-learning algorithm or a reproduced paper result.

What the run shows

End-to-end integration is working

  • PPO and DQN both train and evaluate through the Gymnasium environment.
  • DQN is lowest at 30,000 steps, then worsens; training is not monotonic.
  • PPO finishes lower than DQN at the final checkpoint, but its large seed SD shows instability.
What the run does not show

No universal algorithm ranking

  • Three seeds and untuned settings are not a definitive statistical comparison.
  • TBS contains inventory-specific structure and is not a like-for-like ML method.
  • The demand is synthetic and not calibrated to a company.

Limitations and next steps preserved from the notebook
  • The costs and demand are literature-grounded but synthetic.
  • Three seeds reveal major instability but do not support a definitive ranking.
  • PPO and DQN use reasonable small-run settings without a systematic hyperparameter search.
  • PPO factorizes the two supplier orders while DQN models each complete pair as one action.
  • Stable-Baselines3 one-hot encodes the observation into 2,021 MLP inputs, which may reduce learning efficiency.
  • Result CSV files are saved, but trained model weights are not; policy reproduction requires retraining.
  • A stronger follow-up would use more seeds, longer budgets, separate tuning, and additional inventory heuristics.
  • The scope remains one product and two suppliers; multi-product demand is a separate extension.
